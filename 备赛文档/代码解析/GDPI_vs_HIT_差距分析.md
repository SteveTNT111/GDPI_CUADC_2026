### 2.4 飞行控制精度 🚨

这是**最关键的差距**所在。

| 对比维度 | 哈工大 code7/code9 | GDPI |
|---------|-------------------|------|
| 低通滤波(LPF) | ✅ alpha=0.3 EMA | ❌ 无 |
| 姿态倾斜修正 | ✅ 四元数反投影到地平面 | 依赖TF树（未经姿态补偿） |
| 比例缩放 | ✅ ratio=0.27 | ❌ 无（直接用原始坐标） |
| 远近切换策略 | ✅ 远GPS/近局部 | ❌ 无 |
| 密集航点微分 | ✅ 手动微分 | ❌ 无 |
| 稳定性检测 | ✅ 0.5~1s持续稳定 | 🟡 3s稳定（未实现） |
| better_pos | ✅ 记录最佳位置 | ❌ 无 |
| 摄像头离线检测 | ✅ 100帧超时 | ❌ 无 |
| 高度自适应 | ✅ guidance_norm触发下降 | ❌ 无 |

**分析：**
1. **低通滤波缺失** — 视觉检测输出有抖动（特别是深度图），不加滤波直接用于控制会导致飞机反复修正
2. **姿态倾斜修正缺失** — 飞机倾斜时（前进/刹车/侧移），摄像头不再垂直朝下，计算出的桶位置有系统性偏差。哈工大用四元数将视线向量反投影到地平面来解决
3. **比例缩放缺失** — 哈工大用 ratio=0.27 缩小引导向量，防止过度修正。GDPI 直接使用原始坐标差
4. **better_pos 缺失** — 飞到桶正上方时摄像头可能看不到桶（垂直朝下的盲区），哈工大回到历史最佳位置，GDPI 没有对应策略

**这些差距直接影响投水命中率。** 哈工大北部赛区就是因为这些问题投不进去（"图像存在延迟，因此确定的桶的位置始终不太对"），赛后花了巨大精力才解决。

**1. 实现飞行控制信号滤波链**

```python
# 参考哈工大 code7 的做法
# detector_node 输出 → LPF(alpha=0.3) → 姿态修正 → 比例缩放(ratio=0.27) → 限幅 → 发布
class SignalFilter:
    def __init__(self, alpha=0.3, ratio=0.27):
        self.alpha = alpha
        self.ratio = ratio
        self.filtered = None
    
    def update(self, raw_vector):
        # EMA 低通滤波
        if self.filtered is None:
            self.filtered = raw_vector
        else:
            self.filtered = self.alpha * raw_vector + (1-self.alpha) * self.filtered
        # 比例缩放
        return self.filtered * self.ratio
```

## 五、结论

GDPI 团队通过 AI + web coding 自行构建的代码在**系统架构、视觉检测、坐标变换、文档**等方面已经达到甚至超越了哈工大参考仓库的水平。

但在**飞行控制精度**和**异常处理鲁棒性**这两个决定比赛成绩的核心领域，还有明显的追赶空间。哈工大 2025 年经历了北部赛区的失败后，用一整年的时间迭代出了 code7/code9 中的 LPF 滤波、姿态修正、better_pos、密集航点微分等精细化控制策略——这些是 GDPI 当前代码完全缺失的。

**核心建议：** 把哈工大 code7/code9 中的信号滤波链和异常处理策略用 Python 重新实现到 GDPI 的 geopose_node 和 main.py 中。这不是照搬代码，而是借鉴控制思想——用我们自己的技术栈（Python + ROS1）实现相同的效果。
