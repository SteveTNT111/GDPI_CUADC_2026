> **生效日期**：2026-07-02
> **适用范围**：CUADC + 电赛全体队员
> **核心原则**：代码和知识共用一个仓库。你不写的文档，就是别人明天的坑。

## 一、仓库结构

```
cuadc-2026/                         ← GitHub 仓库根目录
├── README.md                       ← 项目总览 + 5分钟上手指南
├── docs/                           ← 知识文档
│   ├── 系统架构.md
│   ├── 视觉方案.md
│   ├── 控制方案.md
│   ├── 外场测试记录/
│   │   └── 2026-07-02_测试报告.md
│   └── 踩坑记录/
│       └── D435i深度图全黑排查.md
├── vision/                         ← 视觉组代码
│   ├── README.md                   ← 视觉模块部署说明
│   ├── launch/
│   ├── scripts/
│   ├── models/                     ← .pt / .onnx 权重文件（用 Git LFS）
│   └── config/
├── control/                        ← 控制组代码
│   ├── README.md
│   ├── scripts/
│   └── launch/
├── simulation/                     ← 仿真组代码
│   ├── README.md
│   ├── worlds/                     ← .world 文件
│   └── models/                     ← 仿真模型
├── tools/                          ← 工具脚本（日志分析/KML导出等）
└── .gitignore
```

**规则**：
- 一个模块一个文件夹，每个文件夹必须有 README.md
- 权重文件用 Git LFS，不直接提交大文件
- 实验性代码放 `模块名/wip/` 下面，不要污染主目录

## 二、文件命名

| 类型 | 格式 | 示例 |
|------|------|------|
| Python 脚本 | `功能_描述.py`（小写下划线） | `camera_driver.py`、`target_geopose.py` |
| Launch 文件 | `功能.launch` | `vision_all.launch` |
| 测试脚本 | `test_功能.py` | `test_servo_control.py` |
| 配置文件 | `模块.yaml` | `vision_params.yaml` |
| 知识文档 | `简短描述.md` | `深度测距方案.md` |
| 测试记录 | `日期_内容.md` | `2026-07-02_后山试飞.md` |

**禁止**：
- ❌ `最终版.py`、`新版本.py`、`改.py`、`副本.py`
- ❌ 中文文件名（Windows 路径编码坑多）
- ❌ 空格在文件名里

## 三、版本管理

### 必须做

每天收工前：
```bash
git add 你改的文件         # 只加你自己的，不要 git add .
git commit -m "干了什么"   # 一句话说清楚
git push                   # 推到 GitHub
```

每天早上开工前：
```bash
git pull                   # 拉队友的更新
```

### 禁止

- ❌ **绝对禁止** `git add .` —— 会把别人的文件、临时文件、权重文件全加进去
- ❌ 新建文件夹做版本管理（`v1/`、`v2/`、`最终版/`）—— Git 就是干这个的
- ❌ 把代码打成 zip 发到群里 —— 发到 GitHub，告诉组员 `git pull`
- ❌ `git push -f` —— 强制推送会覆盖别人的提交

### Git 不会用怎么办

打开 Claude Code，输入：`教我 git add + commit + push，只教这三个命令`

## 四、交付标准

你交给下一个环节的东西，必须满足：

1. **能跑**：接收方照着 README 敲命令就能运行，不需要找你问"这个参数什么意思"
2. **有 README**：至少包含——
   - 依赖怎么装（`pip install xxx`）
   - 怎么启动（`python3 xxx.py` 或 `roslaunch xxx`）
   - 输入是什么（订阅哪个 ROS 话题）
   - 输出是什么（发布哪个 ROS 话题）
3. **不在本地**：代码在 GitHub 上，不是在你电脑上
4. **没有垃圾**：不要提交 `__pycache__/`、`.vscode/`、`.ipynb_checkpoints/`

### 代码交付格式

```
✅ 好的交付                          ❌ 差的交付
── 功能名/                          ── 桌面/新建文件夹/
   ├── README.md                       ├── 最终版.py
   ├── scripts/                        ├── 最终版_改.py
   │   └── node.py                     ├── 最终版_不改了.py
   └── config/                         ├── test.py
       └── params.yaml                 ├── test2.py
                                       └── 没说明.txt
```

## 五、知识文档

### 什么时候写文档

| 场景 | 写什么 |
|------|--------|
| 装好了某个环境 | `环境搭建_xxx.md`：装的什么、踩了什么坑、怎么解决的 |
| 跑通了一个功能 | 更新对应模块的 README |
| 外场测试 | `测试记录/日期_内容.md`：飞了什么、出了什么问题、日志在哪 |
| 发现了一个坑 | `踩坑记录/问题描述.md`：现象、原因、解决方法 |
| 改了一个参数 | 在对应模块 README 里加一行参数说明 |

### 文档放哪

```
项目文档 → docs/
个人学习笔记 → 05_知识贡献/你的名字/   （留在 Obsidian 里）
踩坑记录 → docs/踩坑记录/
测试记录 → docs/外场测试记录/
```

**个人笔记 vs 项目文档的区别**：
- 个人笔记：写给自己看的，格式随意
- 项目文档：写给队友看的，必须能照着操作

## 六、沟通规范

### 进度同步

每次在群里同步进度，三句话：
```
1. 做了什么：搞定了 D435i 的 ROS 驱动，发布 /camera/color 和 /camera/depth
2. 代码在哪：GitHub vision/ 下面，README 写了怎么跑
3. 下一个做什么：明天开始接 YOLO 检测
```

### 问题求助

问问题附三样东西：
```
1. 你执行的命令或操作
2. 报错信息（截图或复制文字，别只说"报错了"）
3. 你已经试过的方法（证明你不是伸手党）
```

## 八、为什么定这些规则

不是因为"规范很重要"这种空话。

是因为：
- 陈智勇和赖斯远的视觉代码有 306 个 .py 文件，但连他们自己都说不清哪个是主节点
- 控制组写了舵机控制脚本，视觉组不知道在哪、怎么用
- 刘辰宇的仿真代码在他电脑上，他回家了别人没法接手
- 每次交付靠压缩包，版本靠文件夹名

**这些问题不解决，7月21号到赛场发现代码跑不起来，谁都不知道该找哪个文件。**

规范不是为了好看，是为了**你不在的时候别人能接着干**。
